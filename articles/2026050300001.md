# なぜ作成したのか
- 週末に目にした[インシデント](https://corp.moneyforward.com/news/info/20260501-mf-press-1/)が衝撃的だったのでまとめておく 

# はじめに

最近、GitHubを起点とした情報露出・認証情報漏洩系のインシデントが話題になることが増えている。

その中で興味深いのは、

> 「実際の漏洩件数」

以上に、

> 「初期説明への反発」

が大きく炎上するケースがあることだ。

特に今回印象的だったのは、
技術コミュニティとPRメッセージの“温度差”だった。

セキュリティエンジニアやSREは、

> 「それ、かなり危険な類では…？」

と感じる。

一方、初期広報は、

- 限定的
- 支障なし
- 現時点で確認されていない

という比較的冷静なトーンになる。

すると技術者側では、

> 「その温度感じゃないだろ…」

という反応が起きる。

この“認知ギャップ”はなぜ発生するのか。

SREのPostMortemと行動経済学の視点で整理してみたい。

---

# まず整理：「技術事故」と「炎上事故」は別インシデント

ここを分けると、
かなり理解しやすくなる。

今回起きていたのは、

- Incident A：技術インシデント
- Incident B：信頼・炎上インシデント

だった。

しかも後者は、
前者とは別のロジックで拡大していた。

---

# Incident A：技術インシデント

## 何が問題だったのか

技術者コミュニティが強く反応したのは、

> 「GitHubに機微情報・Credentialが存在していた可能性」

だった。

単なるコード流出ではない。

現代クラウド環境ではGitHubは、

- ソース管理
- CI/CD
- IaC
- Automation
- Identity連携

の中心になっている。

つまり、

# GitHub = Production Surface

に近い。

そのためCredential Exposureは、
単なる情報漏洩ではなく、

- 権限昇格
- サプライチェーン侵害
- CI/CD侵害
- 本番横展開

への入口になり得る。

---

## なぜ技術者は敏感だったのか

実務者は、
発表文に書かれていない危険を読む。

たとえば：

- Scopeどこまで？
- Rotation設計は？
- Secret Scan効いてた？
- Push Protectionは？
- 最小権限だった？
- Production分離されてた？
- Runner侵害可能？
- Fork残存は？

など。

つまり、

> 「漏れた」

ではなく、

> 「そこまで到達可能だった」

ことに反応している。

---

## 本当に怖いのは「漏れたこと」ではない

セキュリティ界隈では、
単発ミスより、

- なぜ検知できなかった？
- なぜ防げなかった？
- なぜその権限がそこにあった？

を重く見る。

つまり本当に怖いのは、

> 「見えている漏洩」

ではない。

怖いのは、

> 「その状態が成立していた組織構造」

だ。

これは“個人ミス”というより、

# 境界設計の問題

として受け止められやすい。

---

# Incident B：炎上インシデント

次に起きたのがこちら。

---

# なぜ説明が炎上したのか

興味深いのは、
技術事故そのもの以上に、

# 「説明の温度感」

が問題視されたことだ。

特に危険だったのが：

- 限定的
- 支障なし
- 現時点で確認されていない

系のワード。

企業側は恐らく、

- パニック抑制
- 法務配慮
- 確定情報のみ発信

を意図していた。

しかし技術者側には、

> 「まだ全然わかってないのでは？」

に見えやすかった。

---

# なぜ「限定的」が危険ワードになるのか

危機時、
利用者は“言葉”より、
“意図”を読む。

そのため：

> 「限定的です」

は、

- 小さく見せたい
- ブランド防衛したい
- まだ把握できてない

に変換されやすい。

さらに厄介なのが、

- 限定的
- でも大規模失効
- 限定的
- でもサービス停止
- 限定的
- でも緊急対応

という矛盾。

人間は矛盾情報を受けると、
より危険側に解釈する。

これは認知的不協和（Cognitive Dissonance）として説明できる。

---

# なぜ“まず謝罪”が必要だったのか

企業はつい、
事実説明から始める。

しかし利用者はまず、

> 「不安」

を感じている。

そのため、

- 限定的
- 支障なし
- 確認されていない

より先に必要なのは、

> 「不安を与えたことへの理解」

だった。

危機時、
人は論理より先に、

- 態度
- 誠実さ
- 共感

を見る。

つまり利用者は、

> 「完璧か」

ではなく、

> 「こちら側に立っているか」

を評価している。

---

# なぜ“エンジニア起点炎上”になったのか

ここが今回の特徴だと思う。

通常炎上は、

- 感情
- 印象
- モラル

が中心。

しかし今回は、

# 「技術解説付き炎上」

になった。

つまり：

- SRE
- Security Engineer
- Platform Engineer
- DevSecOps

が、

- 危険性
- 想定侵害経路
- Cloud影響
- Credentialリスク

を解説しながら拡散した。

これが強い。

一般ユーザーは技術詳細がわからなくても、

> 「詳しい人たちがかなり危険視している」

空気を感じ取る。

結果、

- 不安
- 不信
- 疑念

が一気に増幅される。

---

# Incident比較 PostMortem

今回の事案を振り返るうえで興味深いのは、

> 「技術障害」と
> 「信頼障害」

が別々に発生していたことだ。

SREのPostMortem観点で整理すると、
両者はかなり性質が異なる。

| 項目 | Incident A：技術インシデント | Incident B：炎上インシデント |
|---|---|---|
| 障害種別 | セキュリティインシデント | 信頼・認知インシデント |
| 初期トリガー | GitHub上のCredential/機微情報露出 | 初期PRメッセージ |
| 主戦場 | GitHub / Cloud / IAM / CI/CD | SNS / X / 技術コミュニティ |
| 主な影響対象 | システム・認証・運用基盤 | ブランド・信頼・心理 |
| 実害 | Credential Exposure可能性 | 不信感・疑念拡大 |
| 技術者が恐れたもの | Production到達可能性 | 「危機感不足」に見えること |
| 一般ユーザーが恐れたもの | 情報漏洩 | 「本当はもっと危険では？」 |
| 企業側の意図 | パニック抑制・事実整理 | 安心感の提示 |
| 受信側の解釈 | 境界崩壊の可能性 | 保身・矮小化 |
| 危険ワード | Secret / Credential / Production | 限定的 / 支障なし |
| 根本問題 | Boundary設計 | 認知ギャップ |
| 増幅器 | Credential連鎖リスク | SNSアルゴリズム |
| 技術的Lesson | GitHubを本番面として扱う | PRを“心理設計”として扱う |

---

# 一番興味深いポイント

特に印象的なのは、

# 「技術事故の深刻度」と
# 「PRメッセージの危機感」

が一致していないように見えたことだ。

技術者は、
発表文から：

- Cloud権限
- Secret Scope
- CI/CD侵害可能性
- Lateral Movement

を想像する。

つまり、

> 「書かれていない危険」

を読む。

しかしPRは、
一般ユーザー向けに：

- 限定的
- 支障なし
- 現時点で確認されていない

という、
比較的オーソドックスな危機広報を行う。

すると技術者には、

> 「その温度感じゃない」

と映る。

ここで“信頼インシデント”が始まる。

---

# 現代インシデント対応の難しさ

昔のセキュリティ事故は、
比較的「技術問題」で閉じていた。

しかし現在は：

- Cloud Native
- GitHub中心運用
- SNS
- 解説型炎上
- 技術コミュニティ拡散

が存在する。

つまり今は、

# 「技術事故」だけ直しても終わらない。

同時に、

# 「認知事故」
# 「信頼事故」

も処理する必要がある。

---

# 最後に

今回の件で興味深かったのは、

> 「技術対応」と「信頼対応」は別システム

だということだ。

しかもSNS時代では、
後者の障害の方が増幅されやすい。

現代インシデント対応では、

- 技術的正確性
- Secret管理
- IAM
- CI/CD防御

だけでは不十分。

同時に、

- 利用者心理
- 技術コミュニティ認知
- SNS伝播
- 危機広報

まで含めて設計する必要がある。

つまり今のセキュリティ事故は、
単なる「技術問題」ではない。

# “認知と信頼のインシデント”

でもある。
