# IT民俗学：なぜ tmp_final_final2 は生まれるのか

レガシーなサーバーや共有フォルダを見ていると、ときどき奇妙なファイルに出会います。

```text
httpd.conf
httpd_fix.conf
httpd_fix2.conf
httpd_stable.conf
httpd_stable_final.conf
```

怖い。

何が stable だったのか。
何を fix したのか。
なぜ final が存在しているのか。

しかも、不思議なことに、こういうファイルはわりと消せません。

稼働しているファイルは一つ。
他は「もう使ってないはず」とわかっていても、なんとなく残してしまう。

私はこういうファイルを見ると、呪術回戦の七海－真人戦でナレーションされた「触れてはいけない領域がある」という言葉がリフレインします。

## なぜコピーが増殖するのか

もちろん、最初はもっと単純だったはずです。

```text
httpd.conf
```

ただ一つの設定ファイル。

しかし、ある日サーバーが重くなる。

CPU使用率が跳ねる。
Apache のプロセス数が増え続ける。
レスポンスが返らない。

そこで担当者は、**最初のファイルを別名で逃がして**、設定ファイルを少しずつ変え始める。

```apache
KeepAlive On
MaxClients 256
Timeout 30
```

KeepAlive を切る。
MaxClients を変える。
Timeout を短くする。

すると、少し安定する。

しかし別の時間帯になるとまた落ちる。

だから、**今の設定を別名で逃がして**、別の設定を試す。

```text
httpd_fix.conf
httpd_fix2.conf
```

前の方が安定していた気もする。

でも、どれが本当に正しかったのかはわからない。

そこで「**安定していた気がする版**」が保存される。

```text
httpd_stable.conf
```

さらに修正が入る。

```text
httpd_stable_final.conf
```

そして、どこかで「元に戻したい」が発生する。

## 「上書き」が怖い

ここで、ふと思うのです。

なぜ人は、こんなにもコピーを作り続けるのだろう。

現代なら Git がある。
履歴管理もある。
Undo もある。

それでも、人はコピーを作る。

おそらくこれは、単なる技術理解の問題ではないのだと思っています。

たとえば、こんな経験はないでしょうか。

* 上書きして戻せなくなった
* 昨日の状態が必要になった
* 「やっぱり前の版がよかった」が発生した
* 本当に消してよいかわからない

つまり、**「消すこと」が怖い** のです。

## Git以前、ファイルは「世界そのもの」だった

今でこそ、履歴を遡るのは比較的簡単です。

```bash
git diff
git revert
git checkout
```

しかし、昔はそうではありませんでした。

設定ファイルを上書きする。

それはつまり、 **「過去の状態が消える」** ということでした。

しかも当時の環境では、

* observability が弱い
* tracing がない
* rollback が怖い
* 比較検証環境がない
* 本番と同じ負荷を再現できない

ということも多かった。

つまり、 **「なぜ安定したのか」が完全には説明できない** ことがしばしばあったのです。

でも、ある瞬間だけ落ち着く。

レスポンスが返る。
ロードアベレージが下がる。
監視アラートが止まる。

すると、その設定ファイルは急に消せなくなる。

```text
httpd_stable.conf
```

この stable は、ベンチマーク結果というより、 **「誰かがあの夜を生き延びた命綱」** だったのかもしれません。

## 「一時的なコード」が永住することがある

もう一つ、古いコードを見ていると気になるものがあります。

```js
// temporary debug log
console.log("save response", response)
```

一見すると、ただのデバッグコードです。

しかもコメントには「temporary」と書かれている。

でも、妙に消せない。

ふと思うのです。

なぜ、このログは何年も残り続けているのだろう。

おそらく最初は、本当に一時的な調査だったのだと思います。

ある日、本番環境で原因不明の障害が起きる。

再現しない。
ログも足りない。
ローカルでは発生しない。

そこで、とりあえずログを追加する。

```js
console.log("before save")
console.log("response", response)
console.log("after save")
```

次に同じ障害が起きたら、今度こそ原因を追えるように。

しかし、不思議なことに、その後障害は再発しない。

原因はわからないまま。

修正されたのか。
偶然なのか。
別の環境要因だったのか。

誰にもわからない。

ただ、「あの障害があった」という事実だけが残る。

すると今度は、ログを消す理由が失われます。

もし消したあとに再発したらどうするのか。

また原因不明になったらどうするのか。

そうして環境は少しずつ硬直していく。

ログは残る。

コメントも残る。

でも、障害の詳細を覚えている人はいなくなる。

時だけが流れる。

そして最後に、

```js
// temporary debug log
console.log("save response", response)
```

だけが残る。

後から来た人には、理由のわからないコードに見える。

でも、そのコードは、かつて誰かが本番障害に怯えながら追加した **「未来の誰かのために残された命綱」** の名残なのかもしれません。

## 「解除されなかった暫定対応」

こうしてコードを見ていると、レガシーシステムとは「古いコード」そのものではなく、 **解除されなかった暫定対応の積み重ね** によって適者生存した恐竜のようなものではないかと感じます。

| 痕跡           | 背景         |
| ------------ | ---------- |
| final_final2 | rollback恐怖 |
| debugログ      | 原因不明障害     |
| コメントアウト      | 完全削除への不安   |
| retry設定      | 一時的な通信障害   |
| timeout値     | 過去の性能問題    |

どれも最初は、「あとで整理する」つもりだったのかもしれない。

でも、環境が安定してしまうと、逆に触れなくなる。

そして時が経つ。

障害は忘れられる。

背景を知る人はいなくなる。

でもコードだけは残る。

民俗としての祭りや風習も、最初の理由が失われたあと、「なぜか続いているもの」になることがあります。

レガシーコードも少し似ている。

理由は失われる。

でも、「消してはいけない」という感覚だけが残る。

そして、その感覚だけが次の世代に引き継がれていくのかもしれません。

## AI時代、保存場所そのものが変わり始めている

最近は生成AIがコードや文章を書いてくれるようになりました。

私も日常的に使っていますし、本当に便利だと思っています。

ただ、その一方で、少し不思議に感じることがあります。

昔は「途中状態」をファイルとして保存していました。

```text
final
final2
final_final
```


でも最近は、少し様子が違います。

たとえば生成AIとの作業では、

* コンテキストウィンドウ
* Artifact
* Canvas
* チャット履歴

のように、「途中状態」がファイルではなく会話の中に存在することが増えてきました。

以前なら、 **「一旦コピーして残しておく」** だったものが、

今は、 **「この会話ログに残っているはず」** に変わりつつある。

そう考えると、`tmp_final_final2` のような文化は消えたというより、 **「途中状態の保存場所」が変化している** のかもしれません。

## 「消したくない」から「切り離したくない」へ

こうして並べてみると、時代ごとに恐れているものも少しずつ変わっているように見えます。

| 時代    | 恐れていたもの | 保存方法    |
| ----- | ------- | ------- |
| Git以前 | 上書き・消失  | ファイルコピー |
| Git時代 | 差分喪失    | commit  |
| AI時代  | 文脈断絶    | 会話履歴    |

昔は、 **「上書きしたら戻れない」** が怖かった。

だからコピーを残した。

一方、今の生成AI作業では、 **「この文脈を失うと、なぜこうなったかわからなくなる」** 感覚が強くなっています。

だから会話を消せない。

Artifact を残す。

途中生成を保持する。

昔の `tmp_final_final2` が、 **「壊したくない」** の痕跡だったとしたら、今の長いチャット履歴は、**「切り離したくない」** の痕跡なのかもしれません。

## 「変なファイル名」には人間が残っている

最近目にすることが少なくなったレガシーな環境でこういった痕跡ファイルを見つけると、「誰が、どんな気持ちでこれを保存したんだろう」と考えてみるようにしています。

深夜だったのかもしれない。
締切直前だったのかもしれない。
上書き事故があったのかもしれない。

一見すると雑な命名なのに、背景を想像すると急に人間の事情が見えてくる。

頭の中には荒木飛呂彦の『変人偏屈列伝』で描かれた、ウィンチェスター・ミステリー・ハウスの話が展開します。

増築に増築を重ね、どこへ続くのかわからない階段や扉が増えていく奇妙な館。

外から見ると不合理に見える。

でも、当人にはきっと「そうせざるを得なかった理由」があった。

`tmp_final_final2` も同じ構造で生まれたのではないかと想像します。

本来なら整理された構造があるはずなのに、現実には「壊したくない」が優先されて、少しずつ継ぎ足されていく。

そして時間が経つと、理由だけが失われる。

後から来た人には、意味不明な構造に見える。

でも、その奇妙な増築は、かつて誰かが障害や事故と戦った痕跡なのかもしれません。

IT民俗学としては、そういう「理由を失った増築跡」がとても気になります。
