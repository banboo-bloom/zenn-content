# IT民俗学：我々は何に「Hoge」と名付けるのか

ある日、「**テストメールの送信先に `hogehoge.com` を使っていませんか？**」というようなブログ記事を見かけました。

私は思い当たる節があり、内容に目を通しました。

だって、

```text
hogehoge@example.com
hogehoge.com
```

なんて、「存在しないテストアドレス」の代表例のようにテストデータで使用していたからです。

ところが調べてみると、例えば `hogehoge.co.jp` ドメインは実在する。

つまり、存在しないと思い込んでいたドメインも実は存在し、テストメールが本当に誰かへ届いてしまう可能性がある。

そして、本来テストで利用するなら、

```text
example.com
example.net
example.org
```

のような、あらかじめ予約されたドメインを使うことが適切なのだと知りました。

参考：

RFC 2606 - Reserved Top Level DNS Names
https://www.rfc-editor.org/rfc/rfc2606.html

---

「思い込みとは怖いものだ」そう思うのと同時に頭をよぎりました。

> いつからhogehogeが存在しないと錯覚していた？

私は長い間、「Hoge」を存在しないものの代表として扱ってきた。

でも、実際にはそんな保証はどこにもなかったのです。

では、私は一体、「Hoge」に何を期待していたのだろうか。

## 私たちは「Hoge」を教わった記憶がない

新人の頃を思い返してみます。

技術書を読む。

サンプルコードを写す。

先輩のコードを読む。

すると、いつの間にか、

```java
String hoge;
```

```python
foo = "bar"
```

```sql
INSERT INTO users VALUES ('hoge')
```

のようなコードに出会います。

しかし、私は「Hoge」という言葉を誰かに教わった記憶がありません。

オブジェクト指向は教わった。

SQLも教わった。

デザインパターンも教わった。

でも、

```text
hoge
fuga
piyo
```

だけは、いつの間にか知っていた。

どうやら私たちは、概念として学ぶ前に、「Hoge」を身につけていたようなのです。

## 「Hoge」は自得するものではなく、外挿されるもの


私たちの多くは、コーディングを習得する際、

```text
技術書
↓
サンプルコード
↓
模倣
↓
実践
```

という形で学んできました。

つまり、

```text
共同体
↓
サンプルコード
↓
観察
↓
模倣
↓
継承
```

という経路で文化が受け継がれていく。

「Hoge」もその一つなのでしょう。

誰かに定義を教わったわけではない。

先輩が明示的に説明してくれたわけでもない。

ただ、周囲のコードに現れる。

何度も見る。

自分でも使う。

すると、いつの間にか、

> 「こういう場面ではHogeを使うものらしい」

という感覚だけが身についていく。

これは知識の継承というより、共同体の慣習が外挿されている状態に近い気がします。

## メタ構文変数とは何か

実は、このように用いられる「Hoge」には名前があります。

英語では、

> Metasyntactic Variable（メタ構文変数）

と呼ばれています。

参考：

The Jargon File
https://www.catb.org/jargon/html/M/metasyntactic-variable.html

簡単に言えば、「具体的な意味を持たせず、構造だけを説明するための仮の名前」です。

日本では「hoge」がよく用いられますが、海外では

```text
foo
bar
baz
qux
```

などが有名です。

なぜわざわざ意味のない名前を使うのでしょうか。

それは、**本質ではない情報を消すため**です。

例えば、

```java
Customer customer = ...
```

と書くと、コードを読む者は「顧客」のことを考え始めます。

しかし、

```java
Foo foo = ...
```

なら、意識は「顧客」ではなく、アルゴリズムや構造そのものへ向きます。

つまり、

```text
メタ構文変数
↓
意味を消す
↓
構造だけを残す
```

ための道具なのです。

## 我々は何に「Hoge」と名付けるのか

私たちは、「存在しないもの」にHogeと名付けているのではありません。

「まだ意味を与えたくないもの」

「まだ何者か決めていないもの」

「今は構造だけを語りたいもの」

そうした、名指し難いものに対して、

ひとまず、

```text
hoge
```

という印を置いているのです。

例えば、

```java
String hoge;
```

は、まだ何を表す変数か決めていない。

```java
String name = "hoge";
```

は、文字列の中身そのものに意味がない。

```text
hogehoge@example.com
```

は、メールアドレスの形式だけを示したい。

つまり、

```text
hoge
↓
仮の名前
↓
仮の値
↓
仮のデータ
↓
仮のもの全般
```

へと意味が拡張していったように見えます。

本来のメタ構文変数は「仮の名前」を指すものでした。

拡張された「Hoge」は単なるメタ構文変数ではありません。

「とりあえず」

「未定」

「まだ決まっていない」

その全てを表す文化的な記号になっているのです。

## 「Hoge」は存在しないものではなかった

私たちは「Hoge」に、いろいろな期待をしていました。

* 仮のものであること
* 意味を持たないこと
* 本物ではないこと
* 誰にも迷惑をかけないこと

しかし、最後の一つだけは少し怪しい。

```text
hogehoge.com
```

は取得できる。

一方で、

```text
hogehoge@example.co.jp
```

は、むしろ推奨されるテストデータです。

example.co.jp は JPRS が文書・試験用途のために予約しているドメインだからです。

参考：

JPドメイン名の活用について
https://jprs.jp/faq/use/


私たちは慣習的に、

> Hoge = 存在しないもの

だと思っていました。

しかし実際には、

> Hoge = 存在しないと信じられているもの

に過ぎません。

本当に「存在しないこと」が保証されているのは、

```text
example.com
example.co.jp
```

のような、制度として保護された名前の方だったのです。

そして今、同じことをAIも行っています。

大量の技術書。

大量のブログ。

大量のOSS。

大量のサンプルコード。

それらを読み込み、

AIもまた、

```text

```

を学習しています。

考えてみれば、

```text
人間
↓

```

という構造と、

```text
AI
↓

```

という構造は、あまり変わりません。

「Hoge」は、人間だけが知る秘密の言葉ではなくなりつつあります。

人間からAIへ。

共同体からモデルへ。

その文化は、今も静かに継承されているのかもしれません。

## 「Hoge」は、小さな空白の名前だった

私は長らく、「Hoge」は存在しないものの代表だと思って使用していました。

でも実際には違いました。

「Hoge」が存在しないことを保証する仕組みはどこにもありません。

ただ技術者たちが、それを「まだ意味を持たないもの」として扱うことに合意していただけでした。

さながらドーナツの穴のように、そこには何もありません。

でも、その空白があるからこそ、私たちは構造（ドーナツ）について語ることができる。

そう考えると、「Hoge」とは単なる仮の名前ではありません。

まだ名前のないものについて語るために、技術者たちが長い間受け継いできた、小さな空白の名前なのかもしれません。
